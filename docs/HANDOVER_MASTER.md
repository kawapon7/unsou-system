## 1. 開発体制とコラボプロトコル（v3.1）

> **v3.1変更点（2026年7月6日）：** 本番テストが開始しシビアな局面となったため、開発体制を**「Claude Code一本化」**に移行。今後は開発の進行・コード実装をすべてClaude Codeが行う。Cursor・Gemini（ジェミニ）との共同作業は、本番テストが落ち着き再び簡易的な作業フェーズに戻った場合にのみ再開する。旧v3.0の4レイヤー分業（Opus設計→Cursor実装→Sonnet検証）は下記1-3に「簡易作業復帰時プロトコル」として保持する。

### 1-1. チームメンバーと役割（現行・本番テスト期間中）

| メンバー | モデル/ツール | 役割 | 担当領域 |
|---|---|---|---|
| **Atsushi（僕）** | — | 総指揮・ボス | 最終判断・違和感の提示 |
| **Claude Code** | CLI | 設計・実装・検証の一貫担当 | 開発の進行・コード実装・デバッグ・仕様書更新のすべて |

* **Cursor（Composer）・Gemini：** 本番テスト期間中は稼働休止。簡易的な作業フェーズに戻った場合にのみ1-3のプロトコルで再稼働する。

### 1-2. 開発の基本ワークフロー（現行・本番テスト期間中）

```
要望インプット（Atsushi）
  ➔ 設計・実装・検証まで Claude Code が一貫対応
```

複雑なロジック監査が必要な場面では、Claude Code内で `/model claude-opus-4-8` に切り替えて設計精度を優先し、実装・デバッグに戻る際は最新Sonnetモデルに戻す。役割はモデル切り替えのみで完結させ、ツールの分離は行わない。

### 1-3. 簡易作業復帰時プロトコル（本番テスト沈静化後に再開）

本番テストが落ち着き、緊急度の低い簡易作業フェーズに戻った場合のみ、以下の旧v3.0分業体制を再開してよい。

| メンバー | モデル/ツール | 役割 | 担当領域 |
|---|---|---|---|
| **Claude Code（Opus）** | CLI・Opusモード | 最高設計士・ロジック監査 | 複雑な税務・承認ロジックの設計と仕様書の直接更新 |
| **Cursor / Gemini** | IDE | エディタ実装・UI構築 | 複数ファイルにまたがる連続修正・手返しの良い画面開発 |
| **Claude Code（Sonnet）** | CLI・Sonnetモード | 自律デバッグ・テスト実行 | CLIでのバグ修正と検証の自動化 |

```
要望インプット（Atsushi）
  ➔ 【設計フェーズ】設計・仕様書更新（Opus）   ← モデルを Opus に切り替えて投入
  ➔ 【実装フェーズ】UI/エディタ実装（Cursor / Gemini）
  ➔ 【検証フェーズ】CLIデバッグ・テスト実行（Sonnet）  ← モデルを Sonnet に戻して投入
```

**エスカレーションルール（簡易作業復帰時のみ適用）**

* Sonnet が同一問題に **2回試みて解決しない** 場合、ログをそのまま Opus セッションに投入してエスカレーション。
* Opus が方針を確定したら、指示書として Sonnet に戻す。

### 1-4. 絶対ルール

* **結論ファースト・挨拶・前置きの全面禁止**（継続）。解決策・実装方法を最初に述べ、「承知いたしました」等の挨拶や謝罪、まとめの言葉は一切不要。
* **§2-6SのGitコミット前セキュリティ・差分チェックの厳格な順守**（継続）。
* **実動コードの提供**: 原則としてそのまま動作する検証済みコードを出力。環境依存等で検証不可の場合は「未検証」と明記。
* **エラー警告のコメント**: API制限・非同期処理の待機・データ型不一致など、エラーが予想される箇所にはコード内に必ず注意喚起コメントを入れること。

### 2-1. プロジェクト概要

#### 2-1-1. コンセプト
「親分（1人社長）と子分（委託先個人事業主）の両方を支える、運送業専用の業務管理ツール」

#### 2-1-2. ターゲット区分
* **親分（管理者）**
  * 対象：1人社長・個人事業主に業務委託・複数荷主と取引
  * 痛み：月末の計算地獄・請求書作成・支払明細の手作業
* **子分（委託先）**
  * 対象：個人事業主・複数案件を掛け持ち
  * 痛み：売上管理・運行記録の不安

#### 2-1-3. ビジネスモデル
* サブスクリプション型（月額課金）
* 子分アプリの費用は親分負担（子分は無料で利用）
* フィールドテスト：知り合いの1人社長から開始（激安モニター価格）
* 展開戦略：親分を落とせば子分への半強制展開が可能

#### 2-1-4. 料金設計
* **基本構造**
  * 基本料金：2,980円/月（親分1アカウント）
  * 登録者数：980円/人/月
  * 子分アプリは親分負担 → 子分は無料で利用できる
* **カウントルール**
  * 当月1日でも稼働記録があれば1人としてカウント
  * 締め日：月末
  * 退職＋新規入れ替え＝2人分カウント（マスターデータが増えるため）
  * 例：5人稼働中に1人退職・1人新規入社 → その月は7人分（5＋退職1＋新規1）
* **料金シミュレーション**
  * 子分3人：5,920円（2,980＋980×3）
  * 子分5人：7,880円（2,980＋980×5）
  * 子分10人：12,780円（2,980＋980×10）
  * 5人＋途中交代2人：9,840円（2,980＋980×7）

#### 2-2-2b. デプロイ構成（Cloudflare Workers｜2026-07-01〜）

> **重要：Pages から Workers に移行済み。** `@opennextjs/cloudflare` v1.20.1 は **Workers 専用アダプタ**で、Pages 形式（`_worker.js` ディレクトリ）を出力しない。Pages で運用していたため全ルート404になっていた。

* **本番URL**：`https://unsou-system.kawapon7.workers.dev`（旧 `*.pages.dev` は404のまま）
* **デプロイ**：`web/` で `npm run deploy`（＝`ALLOW_DEV_AUTH_BYPASS=false opennextjs-cloudflare build && opennextjs-cloudflare deploy`）。要 `CLOUDFLARE_API_TOKEN`（Workers Scripts:Edit）＋`CLOUDFLARE_ACCOUNT_ID`。
* **ローカルプレビュー**：`npm run preview`（＝`opennextjs-cloudflare build && wrangler dev`）
* **設定**：`web/wrangler.toml` に `main=.open-next/worker.js` と `[assets] directory=.open-next/assets, binding=ASSETS`。
* **シークレット**：`wrangler secret put <NAME>` で Worker に投入（NEXT_PUBLIC_SUPABASE_URL/ANON_KEY/SUPABASE_SERVICE_ROLE_KEY/ENCRYPTION_KEY/GEMINI_API_KEY/RESEND_API_KEY）。Pagesダッシュボードの変数は引き継がれない。
* **自動デプロイ**：旧Pages連携は無効。main push 時の自動化は Workers Builds の再設定が必要（未対応）。
* **ビルド前提**：`useSearchParams()` 使用ページは `<Suspense>` 境界必須（無いと `next build` が失敗し worker 未生成→404）。admin配下は `admin/layout.tsx` で一括Suspenseラップ＋`force-dynamic`（認証ガードのバイパス防止）で対応済み。

#### 2-2-6. 拡張オプション連携設計方針（v1.9 rev.1追加）

将来的に独立した拡張オプション（VOICE / SCAN 等）とプラグイン形式で連携するため、以下の設計方針を採用する。

* **疎結合（プラグイン型）アーキテクチャ**：拡張オプションはHIBIKIのコアDB設計・計算ロジック・RLSポリシーを一切改変しない。各オプションはHIBIKIが提供する内部APIエンドポイント（`/api/hibiki/` 配下）を通じてのみデータの読み書きを行う。
* **APIインターフェース**：オプション側からSupabase DBへの直接アクセスを禁止し、Server Actions / API Route 経由でのみ最小権限を付与する構造を維持する。
* **フック設計**：主要な業務イベント（勤務記録登録・立替金登録・請求書確定等）にフックポイントを設け、オプション側が後処理を差し込めるコールバック構造を用意する。
* **データ拡張用フィールド**：`work_records`・`expense_records`・`clients` テーブルに `metadata JSONB` カラムを追加マイグレーションで予約し、各オプションが名前空間付きで付加情報を保存する（例：`{"voice": {...}, "scan": {...}}`）。
* **セキュリティ継承**：拡張オプション側は本節 2-2-7 で定めるセキュリティ要件（AES-256-GCM暗号化・RLS全面遮断）をすべて継承・厳守する。

#### 2-2-7. セキュリティ要件（v1.9追加：個人情報・口座情報の最終防衛ライン）

##### 2-2-7-1. 口座情報の暗号化保存（必須）
* 口座情報（銀行名、支店名、口座種別、口座番号、口座名義）は、DB（Supabase）への**平文保存を厳禁**とする。
* Next.jsのサーバーサイドで、環境変数 `ENCRYPTION_KEY`（32バイトの秘密鍵）を使用した **AES-256-GCM方式** により暗号化してDBへ保存する。
* 取得時にのみサーバーサイドで復号し、画面（Reactコンポーネント）に渡す構造とする。ブラウザに暗号文が届くことはない。
* 暗号化ユーティリティは `utils/crypto.ts` に集約する（`encryptText` / `decryptText`）。

##### 2-2-7-2. RLSによるDB直接アクセスの全面禁止
* 口座情報を含むテーブル（`contractors`、`clients` 等）へのブラウザからの直接アクセス（SELECT / INSERT / UPDATE）は、SupabaseのRLS（行単位セキュリティ）により**全面禁止**とする。
* サーバーサイド（API Route / Server Actions）経由のみを許可する。
* `anon` ロールはすべてのテーブルへのアクセスをデフォルト拒否とし、`authenticated` ロールに最小権限のみ付与する。

##### 2-2-7-3. 環境変数の管理
* `ENCRYPTION_KEY`（32バイト） は `.env.local` にのみ保持し、Gitリポジトリへのコミットを禁止する（`.gitignore` で管理）。
* 本番環境（Cloudflare Workers）では `wrangler secret put <NAME>` で Worker のシークレットとして注入する（Pagesダッシュボードの変数は移行後は使われない）。

#### 2-3-3. AIスキャン（`/admin/scan`）

* **IN（売上書類取込）**：荷主からの発注書・受領書をアップロード → Gemini API が解析 → 荷主マスタと照合 → `invoices` テーブルへ `status='draft'` で保存（`saveClientScanResult` Action）

* **IN/OUT 共通フロー**：ドロップゾーン → アップロード → ポーリング（3秒間隔・最大40回） → レビューフォーム（修正可） → 確定・保存
* **実装ファイル**：`web/src/app/admin/scan/page.tsx` / `web/src/app/_actions/scan-actions.ts`

#### 2-3-5. 消費税・インボイス計算ロジック
* **基本方針**
  * DBの明細行には税抜き金額のみ保存
  * 消費税額は請求書・支払通知書の確定タイミングで一括計算
  * 税込み請求が来た場合は自動で税抜きに変換して保存
  * 立替金は実費金額で保存し、消費税区分を別途管理
* **税込み→税抜き変換**
  * 税抜き金額 = 税込み金額 ÷ 1.1
  * 消費税額 = 税込み金額 × 0.1 ÷ 1.1
  * 端数処理 = 四捨五入（v1.6統一）
* **消費税の端数処理タイミング（v1.6更新）**
  * 【明細行（work_records / project_payees）】→ 税抜き金額のみ保存（消費税額はここでは計算・保存しない）
  * 【請求書・支払通知書の確定時】→ インボイス区分ごとに明細を分類 → 区分ごとに税抜き金額を合計 → 合計に対して消費税を一括計算 → 端数処理（1請求・税率ごとに1回、四捨五入）を実施 → 確定した消費税額をテーブルに保存
  * 理由：明細ごとに端数処理すると、合計時に1円単位のズレが発生するため（インボイス制度厳守）
* **インボイス未登録業者への支払い計算**
  * 経過措置による仕入税額控除の控除不可分を支払金額から差し引く。システムは取引日から控除率を自動判定・自動切替する。
  * 〜2026年9月30日：控除可能割合80%／控除不可割合20%／差し引き率2%（消費税10%×20%）
  * 2026年10月1日〜2029年9月30日：控除可能割合50%／控除不可割合50%／差し引き率5%（消費税10%×50%）
  * 2029年10月1日〜：控除可能割合0%／控除不可割合100%／差し引き率10%（消費税10%×100%）
* **インボイス区分ごとの消費税集計フロー**
  ```
  明細行を3カテゴリ（登録あり／未登録／非課税）に分類
    ↓
  カテゴリごとに税抜き金額を合計
    ↓
  カテゴリごとに消費税を一括計算・端数処理（四捨五入1回）（v1.6更新）
    ↓
  カテゴリごとの小計を合算してトータルを算出
    ↓
  確定した金額・消費税額をスナップショットとして保存
  ```

#### 2-3-12. お気に入り（ショートカット）機能（v1.9.1追加）

##### 概要
親分アプリの任意の画面・タブを「⭐️」ボタン1つでサイドバーにピン留めし、次回から1タップで直接その状態に戻れるショートカット機能。

##### 星ボタンの配置
* **デスクトップ**：左サイドバーのロゴ（HIBIKI）の右隣に常時表示
* **モバイル**：画面上部ヘッダーの右側（ログアウトボタンの左）に常時表示
* **管理対象外ページ**（認証画面等）では星ボタンは自動で非表示になる
* 各 `page.tsx` 側にボタンを持たせる個別実装は廃止。`web/src/app/admin/shell.tsx` への完全一元化により、今後メニューを追加しても `PAGE_DEFS` に1行追記するだけで自動対応。

##### URL記憶仕様
* 星を押した瞬間のURLクエリパラメータを含むフルURL（例：`/admin/sales?tab=scan`）を記憶する。
* タブ・フィルターはすべてURLパラメータとして管理しているため、保存したURLに遷移するだけで対象タブが復元される。
* 各ページのパラメータキー：

| ページ | パラメータキー | 値の例 |
| :--- | :--- | :--- |
| 案件・配車管理 | `?status=` | `in_transit`（運行中）等 |
| 売上管理 | `?tab=` | `scan`（AIスキャン入力）等 |
| 取引先マスタ | `?tab=` | `clients`・`contractors` |
| 請求・支払管理 | `?tab=` | `billing`・`payment` |

##### ラベル自動生成
* `shell.tsx` 内の `PAGE_DEFS` マップがページ名とタブ名の対応を管理する。
* 星を押した時点のパラメータを読み取り、`「ページ名（タブ名）」` の形式でラベルを自動生成して保存する。
  * 例：`?tab=scan` → `売上管理（⑥ AIスキャン入力）`
  * 例：`?status=in_transit` → `案件・配車管理（運行中）`
  * パラメータなし（デフォルト表示）→ `売上管理` のみ
* 新しいページ・タブを追加する際は `PAGE_DEFS` に1エントリ追記するだけで対応完了。

##### データ永続化（localStorage）
* お気に入りデータは `localStorage`（キー：`hibiki_admin_favorites`）に JSON 配列として保存する。
* React 18 の `useSyncExternalStore` を使用して購読・更新を管理（`useState + useEffect` パターンは採用しない）。
  * `getServerSnapshot` は安定した空配列定数を返す（無限ループ防止）
  * `getSnapshot` は文字列比較によるキャッシュで参照安定性を保証
  * 同一タブ内の複数コンポーネントは `hibiki:favorites-change` カスタムイベントで同期
* ブラウザリロード後もデータは保持される。

##### 誤操作防止
* サイドバーのショートカット一覧にある「✕」削除ボタンを押すと、`window.confirm` により `「[メニュー名]」をお気に入りから削除しますか？` の確認ダイアログが表示される。
* OK で削除実行、キャンセルで中断。
* 星ボタン自体のトグル（登録・解除）は快適性優先のため確認ダイアログなしで即時動作。

#### A. 防衛的設計原則（サイレントトラブル自動検知）

人間の目が離れたことで起きるサイレントな大破綻（入力ミス・二重請求・マスタ失効）を自動検知・遮断する思想をコアに組み込む。

* **入力遅延の自動検知：** `schedules`テーブルの予定ステータスと`work_records`の実績を突合し、入力漏れを無人で発見してアラートを起票する。
* **二重請求の自動遮断：** 同一ドライバー・同一日・同一案件の複数実績を登録時点で検知し、ブロックまたは確認モーダルへ強制誘導する。
* **マスタ失効の自動検証：** インボイス登録番号の有効性を請求書確定前に自動検証し、失効検知で経過措置（控除率自動切り替え）へ強制移行する。
* **業務しきい値ガード：** 個数100超・立替金3万超等の異常値を登録時に自動ロック（`'pending_review'`ステータスへ移行）し、親分の明示的確認なしに確定させない。
* **長期未承認の自動検知：** 支払通知書の送信後48時間以上承認なしの状態を定期ジョブで監視し、アラートを起票する。

**インポートフロー**
1. 子分：入力専用Googleフォームに稼働実績を入力（自分の入力画面のみ）。
2. 親分：「緊急インポート」画面でGoogleスプレッドシートURLまたはCSVを読み込む。
3. パース：`fileConverter.ts`（SCANオプションと共通モジュール）がファイルをパースし列ヘッダーを自動認識。
4. マスタ照合：案件名・ドライバー名等をマスタデータとファジーマッチングし表記揺れを自動吸収。
5. 手動補正UI：信頼スコア閾値以下の行を一覧表示し、親分が正しいマスタ値を選択・補正。
6. 一括登録：`importEmergencyRecords`（`web/src/app/_actions/workRecordActions.ts`）経由で `createWorkRecord` を順次呼び出し登録。

**セキュリティ制約**
* 本機能の実行は親分権限のみ（子分ロールはRLSで全面拒否）。
* Googleフォームは子分に入力専用URLのみ共有し、集約スプレッドシートは親分のみ秘匿。子分同士のデータ閲覧を100%遮断する。

### 3-1. 概要・目的

#### コンセプト
運転中・現場作業中・両手がふさがっている状況でも、スマートフォンのマイクに話しかけるだけでシステムの操作・データ入力を完結させる「ハンズフリー業務入力」オプション。

子分（ドライバー）が走行中でも記録を残せるようにすることで、入力遅延・漏れをゼロに近づけ、月末の記憶頼り入力を根絶する。

#### ターゲットユーザー

| ユーザー | 主要ユースケース |
| :--- | :--- |
| ドライバー（driver UI） | 稼働終了時に音声で「今日の記録を登録して」「高速代1,200円を立替に追加」 |
| 管理者（admin UI） | ハンズフリーでダッシュボード確認・画面遷移 |

#### 設計原則
* HIBIKIコアのDB・ロジック・RLSポリシーを**一切改変しない**疎結合設計
* 音声認識処理はフロントエンド（ブラウザ Web Speech API）で完結させ、テキスト化されたインテントのみをHIBIKI APIへ送信する
* 音声データ（音声ストリーム）をサーバーへ送信・保存しない（プライバシー保護）
* HIBIKIコアのセキュリティ要件（AES-256-GCM暗号化・RLS全面遮断）を完全継承

#### V-04: 案件・取引先情報サジェスト
* 案件登録フォームで「城南エリア」と話すとマスタとファジーマッチし候補を表示
* 取引先登録フォームで「会社名を音声入力」し既存マスタと照合
* 音声入力後のフォーム内容は必ず視覚的確認ステップを挟む（音声のみでの確定禁止）

#### V-05: ダッシュボード読み上げ
* 「今月の状況を教えて」等のコマンドで、ダッシュボードのサマリーを音声合成（Web Speech Synthesis API）で読み上げる
* 読み上げ内容：入金予定総額・支払予定総額・粗利・未承認件数

| HIBIKIコア要件 | VOICEオプションの実装方針 |
| :--- | :--- |

* **変更履歴：**

### 4-1. 概要・目的

#### コンセプト
紙媒体のスキャン画像・PDF・Googleスプレッドシート・Excelなど、多種多様な形式で外部から送られてくる請求書のデータを自動抽出し、HIBIKIのマスタと照合しながら入力業務を**半自動でサポート**する拡張オプション。

**完全自動化ではなく「人間が最終確認・承認する半自動設計」を徹底する。** AIが抽出・提案し、人間が判断・確定するUX。

#### 解決する課題

| 現状の課題 | SCANオプションによる解決 |
| :--- | :--- |
| 荷主から届く請求書フォーマットが各社バラバラ | AIが多様な形式を統一的に解析・構造化データとして抽出 |
| 手入力による転記ミス・時間ロス | 抽出結果をフォームにプリセットし確認ステップで精度補完 |
| 未登録取引先の手動照合作業 | マスタ自動照合＋新規登録ナビで漏れなく対応 |
| 大量件数処理時のシステム負荷 | 非同期キュー設計で将来の大量処理に対応 |

#### 設計原則
* HIBIKIコアのDB・ロジック・RLSポリシーを**一切改変しない**疎結合設計
* SCANオプション側からSupabase DBへの直接書き込みは**厳禁**とし、必ずHIBIKI APIを経由する
* HIBIKIコアのセキュリティ要件（AES-256-GCM暗号化・RLS全面遮断）を完全継承
* AIは「提案者」に徹し、データの確定は必ず人間が行う

### 4-2. 対応入力形式

| 形式 | 対応方法 | 備考 |
| :--- | :--- | :--- |
| 紙スキャン画像（PNG / JPEG） | Gemini Vision APIで画像解析 | スマホカメラ撮影も可 |
| PDF | PDF→画像変換後にVision API解析 | テキストレイヤー有無に関わらず対応 |
| Googleスプレッドシート | Google Sheets APIでセル値取得後に構造解析 | 共有URL貼り付けで取込 |
| Excel（.xlsx） | SheetJS等でパース後に構造解析 | ファイルアップロード形式 |
| CSVテキスト | テキストパース後に構造解析 | コピー&ペースト入力も可 |

### 4-3. 技術アーキテクチャ

#### 全体アーキテクチャ図

```
[管理画面ブラウザ（admin）]
  │ ファイルアップロード / URL貼り付け（/admin/sales?tab=scan）
  ↓
[SCAN APIエンドポイント]（/api/scan/upload）
  │ ファイルを受取り、ジョブIDを即時返却（非同期分離）
  ↓
[ジョブキュー]（Cloudflare Queues / Supabase Edge Functions等）
  │ 非同期でAI解析ジョブを実行
  ↓
[AI解析モジュール]（Gemini 2.0 Flash API）
  │ 請求書データを構造化JSONとして抽出
  ↓
[マスタ照合モジュール]（HIBIKI Server Actions 経由）
  │ 取引先名・案件名をHIBIKIマスタと照合
  ↓
[サジェスト結果DB保存]（scan_jobs テーブル）
  ↓
[管理画面ブラウザ（admin）]
  │ ジョブIDでポーリング or WebSocket通知
  ↓
[確認・承認UI]（ScanTab.tsx：サジェスト表示→人間が承認→Server Actions へ確定送信）
  ↓
[HIBIKI コアDB]（RLS保護下・service_role経由のみ）
```

### 4-4. 業務フロー・マスタ照合ロジック

#### 全体フロー

```
① ファイルアップロード
  ↓
② AI解析・データ抽出（非同期）
  ↓
③ 取引先マスタ照合
  ├── 【一致あり】→ 自動紐付け（④へ）
  └── 【一致なし】→ 新規登録ナビ（⑤へ）
  ↓
④ 案件マスタ候補サジェスト（既存データあり）
  ↓
⑤（新規のみ）新規登録フォームに自動プリセット
  ↓
⑥ 人間による確認・承認
  ↓
⑦ 確定データをHIBIKI APIへ送信
  ↓
⑧ HIBIKIコアDBに反映（invoices テーブル等）
```

#### 取引先マスタ照合ロジック

```
AIが抽出した issuer_name
  ↓
HIBIKI API GET /api/hibiki/clients?search={issuer_name} で照合
  ↓
一致スコアを計算（完全一致 > 部分一致 > 読み仮名一致 > 類似度スコア）
  ↓
スコア閾値（例：80%以上）で自動紐付け判定
  ↓
80%未満 → 候補リストをUI表示してユーザーが選択
```

**【既存データあり】の場合**
```
取引先「〇〇運輸株式会社」← 既存マスタと一致（スコア95%）
  ↓
[自動紐付け済み] 表示
  ↓
関連する進行中の案件マスタ候補をプルダウンでサジェスト：
  ● 城南エリア宅配便（2026年5月〜）
  ● 企業間スポット配送（2026年4月〜）
  ↓
ユーザーが案件を選択 → 請求金額・明細を確認 → 「確定して取り込む」ボタン
  ↓
POST /api/hibiki/invoices（確定データ送信）
```

**【既存データなし】の場合**
```
取引先「新規物流センター株式会社」← マスタに存在しない
  ↓
[未登録の取引先「新規物流センター株式会社」が検出されました。新規登録しますか？]
バナーをUI上部に表示
  ↓
「新規登録する」ボタン押下
  ↓
```

### 4-5. 非同期処理（キュー管理）アーキテクチャ

#### 設計方針
**アップロード処理とAI解析処理を明示的に分離する。** これにより：
* 大量ファイルアップロード時でもUIがブロックされない
* AI解析の遅延・失敗がユーザー体験を損なわない
* 将来的な100件以上の同時処理に対応できる

#### 処理フロー（非同期）

```
[フロントエンド]
  POST /api/scan/upload（ファイル送信）
    ↓ 即時レスポンス
  { jobId: "scan_abc123", status: "queued" }
    ↓ UIは「解析中...」状態に遷移
  
[バックエンド - ジョブキュー]
  scan_jobs テーブルに INSERT（status: "queued"）
    ↓ 非同期ワーカー起動
  Gemini API呼び出し（数秒〜数十秒）
    ↓ 完了後
  scan_jobs テーブルを UPDATE（status: "completed", result: {JSON}）

[フロントエンド]
  GET /api/scan/jobs/{jobId}（ポーリング or WebSocket）
    ↓ status: "completed"
  確認・承認UI表示
```

#### scan_jobs テーブル定義（追加マイグレーション予定）

```sql
CREATE TABLE scan_jobs (
  id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  uploaded_by     UUID        NOT NULL REFERENCES users(id),
  file_type       TEXT        NOT NULL,  -- 'image', 'pdf', 'spreadsheet', 'csv'
  file_url        TEXT        NOT NULL,  -- Supabase Storageの一時URL
  status          TEXT        NOT NULL DEFAULT 'queued',
                              -- queued | processing | completed | failed
  ai_model        TEXT        NOT NULL DEFAULT 'gemini-1.5-flash-latest',
  extracted_data  JSONB,                 -- AI抽出結果
  confidence_score NUMERIC(4,3),         -- AI信頼スコア（0.000〜1.000）
  matched_client_id UUID      REFERENCES clients(id),
  error_message   TEXT,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### スケーリング戦略

| 処理件数/月 | 推奨構成 | コスト目安 |
| :--- | :--- | :--- |
| 〜20件 | Supabase Edge Functions（同期処理）| ほぼ無料 |
| 20〜100件 | Edge Functions + scan_jobs テーブルによるキュー管理 | Gemini Flash APIコストのみ（数百円/月） |
| 100件以上 | Cloudflare Queues + Workers / BullMQ 等の本格キューイング | 要コスト計算 |

#### 2026-07-18（Claude Code セッション・分析のみ、未着手）

**一般的な請求・支払い管理ソフト（freee／マネーフォワード／弥生等）と比較したHIBIKIの機能ギャップを分析。実装は未着手、次回以降の検討課題として記録。**

現状の実装内容（§2-3-3〜2-3-6の請求書生成・支払通知書・立替金・インボイス税務・収支ビュアー）を踏まえ、一般的な請求・支払い管理ソフトに対して概念自体が存在しない・手薄な領域は以下7点：

1. **電子帳簿保存法対応（優先度高）**：現行仕様は「PDFはその都度生成方式（DBに保存しない）」（§2-2-3）。計算ロジックが将来変わると過去の請求書を再現できなくなるリスクがあり、税務調査対応の観点で要検討。
2. **入金消込の自動化（優先度高）**：現状は「入金確認ボタン」による手動ステータス更新のみ（§2-3-3画面③）。銀行明細CSV取込・API連携による自動突合の概念がない。荷主数増加で手動確認が破綻しやすい。
3. **会計ソフトとの仕訳連携（優先度高）**：確定した税抜き金額・消費税額をfreee/マネーフォワード/弥生へエクスポートする仕組みがなく、記帳が別途手作業になっている可能性。
4. **督促・延滞管理**：入金予定日超過を検知した自動督促の概念が見当たらない（未入力アラートはあるが未入金アラートは無さそう）。
5. **与信管理**：委託先・荷主ごとの支払遅延履歴や与信枠を追跡する仕組みがない。
6. **見積書機能**：請求書・支払通知書はあるが、見積→承認→受注→請求という前段のワークフローがない。
7. **支払データの自動生成（全銀フォーマット）／取引先セルフサービスポータル**：支払通知書はPDF出力止まりで銀行振込用一括データ生成がなく、荷主・委託先が自分でログインして請求書を確認できるポータルも無い（子分アプリはあるが荷主側は未確認）。

### 5-8. 認証・権限のルール

- `user_metadata.role = 'master'` → admin（親分）
- `user_metadata.role = 'driver'` → driver（子分）
- `user_metadata.tenant_id` → テナント識別。**dev 環境は `'local-dev'` 固定**
- ログイン後: admin → `/admin/dashboard`、driver → `/driver/schedule`
- DBアクセスは必ず **サービスロール** 経由（RLS回避）＋ `tenant_id` or `contractor_id` でフィルタ

#### ユーザーとドライバーの紐づけ方法（重要）
`public.users` テーブルに `contractor_id` カラムは**存在しない**。
紐づけは `contractors.email = auth.users.email` のメール一致で行う。
- ドライバーアカウント作成時: `contractors.email` に新アカウントのメールを書き込む
- ドライバーログイン後: `contractors WHERE email = 自分のメール` でcontractor_idを取得
- `users/actions.ts` の `createDriverUser` / `updateUser` がこのロジックを実装

### 5-9. やってはいけないこと

- `anon` ロールでDBに直接アクセスしない（RLS で全面禁止）
- 口座情報を平文でDBに保存しない（必ず `crypto.ts` で暗号化）
- `approval_history` / `notification_logs` に UPDATE / DELETE を発行しない（不変ログ）
- `ENCRYPTION_KEY` を Git にコミットしない
