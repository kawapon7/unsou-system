# HIBIKI開発 効率化スキル 設計書

- **日付**: 2026-07-23
- **目的**: HIBIKI開発全体の「時間短縮」と「トークン削減」を、安全性を犠牲にせず両立する。
- **背景**: システムが厚くなるにつれ、Claude Code の思考時間・作業時間が増大。実測で確認した2大ボトルネックは **①仕様書（HANDOVER_MASTER.md 1749行 / 132KB ≒ 33kトークン）の全文読み込み** と **②思考・計画の長さ（小さな作業にも過大な手順）**。

## 優先順位（絶対）

1. **安全第一**（セキュリティ規約の遵守は効率化より常に優先）
2. その上での **時間・トークンの節約／効率化**

効率化が安全と衝突する場合は、必ず安全を採る。

---

## 全体構成：2スキル体制

| スキル | 役割 | 発動 |
|---|---|---|
| **hibiki-dev**（入口・交通整理） | 労力のライトサイジング＋仕様ナビ。実装そのものはしない | 自動（開発タスク冒頭）＋手動 `/hibiki-dev` |
| **hibiki-security**（軽量ゲート） | セキュリティ確認の入口。規約本文は複製せず、判定と参照先のみ | hibiki-dev の判定 or git操作トリガー時のみ（自動発動しない） |

- 配置: `.claude/skills/hibiki-dev/SKILL.md` / `.claude/skills/hibiki-security/SKILL.md`
- **軽量性の数値上限**（歯止め）: hibiki-dev ≦100行、hibiki-security ≦60行。スキル自身の読み込みが浪費源にならないよう縛る。

---

## スキル1: hibiki-dev

### 機能A: 労力のライトサイジング（②思考・計画の長さ対策）

タスク冒頭で4段階に即分類し、**一言で宣言してから**着手する。

| 段階 | 目安 | 手順 |
|---|---|---|
| 些細 (Trivial) | 1ファイルの機械的編集、typo、文言、定数変更 | 長考・プロセススキルなしで直行。仕様ナビ不要 |
| 小 (Small) | 数ファイル・既存パターン踏襲で完結 | 該当仕様節だけナビ取得→直行。プロセススキル起動しない |
| 中 (Medium) | 新規機能の一部、複数コンポーネント連携、非自明ロジック | 仕様ナビ＋必要なプロセススキル（systematic-debugging 等） |
| 大 (Large) | 新機能まるごと、複数サブシステム、DB設計変更 | brainstorming → writing-plans をフル起動 |

**機械的な先行チェック（段階判定より前に評価）**:
- **口座情報・DBアクセス・不変ログ（`approval_history`/`notification_logs`）を1文字でも含むなら、自動的に最低「中」扱い＋ hibiki-security を起動。** 主観判定による漏れを防ぐ。

**安全弁（CLAUDE.md「最速より確実」の維持）**:
- 迷ったら重い側に倒す。
- バグ・不具合の報告は段階に関係なく systematic-debugging を通す。

### 機能B: 仕様ナビ・プロトコル（①仕様書読み込み対策 / 本丸）

**目標**: HANDOVER_MASTER.md 全文読み（≒33k）を廃止し、目次→該当節だけの **3〜5k** に圧縮（1/7〜1/10）。

**手順（4ステップ）**:
1. **全文読み禁止**。`Read` で HANDOVER_MASTER.md を offset/limit なしに開かない。
2. **目次だけ取得**（最初の一手）:
   ```
   grep -nE '^#{1,3} ' docs/HANDOVER_MASTER.md
   ```
   見出し＋行番号のみ（≒1k弱）。
3. **該当節を特定** → 下記ショートカット表（節番号で指す）。表になければ手順2の目次から判断。
4. **ピンポイント読み**: 目次の行番号から「該当節の開始行〜次の見出しの行」を割り出し、`Read` の offset/limit でその範囲だけ取得。

**陳腐化しない設計（肝）**:
- スキルには**行番号を一切書かない**。安定した**節番号(§)のみ**を持つ。行レンジは手順2の grep で**都度動的に解決**。→ 作業履歴（§5-4）が伸びて行番号がずれてもスキルは壊れない。

**ショートカット表（タスク種別 → 節）**:

| やりたいこと | 見る節 |
|---|---|
| 現状把握・次に何をやるか | §5-2, §5-3, §5-4 |
| どのファイルを触るか | §5-5 ファイルマップ（最頻の起点） |
| Server Action を書く/直す | §5-6 |
| admin画面の仕様 | §2-3 |
| ドライバーUI | §2-4 |
| データ設計・テーブル | §2-5, §5-7 |
| 認証・権限ルール | §5-8 |
| 音声操作 | §3 |
| 請求書取り込み | §4 |
| 環境変数・起動 | §5-10, §5-11 |

（セキュリティ §2-6S は hibiki-security 側が保持）

**さらなる節約ルール**:
- 同一セッションで一度読んだ節は再読しない（会話に残っている＝タダ）。
- 些細タスクは仕様ナビ自体スキップ。
- 節を読んで足りなければ隣接節を1つずつ足す（最初から広く読まない）。

## スキル2: hibiki-security（軽量ゲート）

**原則**: 規約本文を**複製しない**（CLAUDE.md / HANDOVER §2-6S との三重管理・更新漏れを防ぐ）。持つのは「①発動トリガー ②最小チェックリスト ③参照先ポインタ」のみ。

**発動トリガー**（いずれかに触れたら発動）:
- 口座情報の読み書き・表示・保存
- DBアクセスを書く/変える（Server Action・RLS・クライアント直クエリの疑い）
- `approval_history` / `notification_logs` を扱う
- **git commit/add に進む直前**（最頻・最重要）
- `.env.local` / デプロイ設定 / 認証フラグに触れる

**チェックリスト（判定と参照先のみ）**:

| # | 確認 | 詳細参照 |
|---|---|---|
| 1 | 口座情報を平文でDBに入れていないか（`utils/crypto.ts` の AES-256-GCM 経由か） | CLAUDE.md §2 |
| 2 | クライアント直クエリになっていないか（全DBアクセスは Server Actions 経由か） | CLAUDE.md §2 |
| 3 | `approval_history`/`notification_logs` に UPDATE/DELETE を書いていないか（INSERTのみ） | CLAUDE.md §2 |
| 4 | **コミット前3ステップ**: `git status` で `.next/`・`.open-next/` 未混入 → `git diff --cached \| grep -E "SERVICE_ROLE_KEY\|GEMINI_API_KEY\|RESEND_API_KEY\|ENCRYPTION_KEY"` が空 → ファイル明示 add | HANDOVER §2-6S |
| 5 | `.env.local` の挙動フラグをビルドに焼き込んでいないか（deploy時 `ALLOW_DEV_AUTH_BYPASS=false` 強制） | HANDOVER §2-6S |

**運用**: 該当項目だけ即チェックして「OK/要対応」を一言宣言。全項目を毎回精読しない。**git 操作時は #4 を必ず通す（スキップ禁止）。**

## CLAUDE.md の修正

現行の「superpowersスキルが1%でも適用可能と判断した場合は必ず呼び出し」を、以下へ書き換える:

> hibiki-dev / hibiki-security を含む各スキルを対等に扱い、状況に応じて最適と思われるスキルを選んで使用する。

- using-superpowers の強制起動も、この新方針（最適選択）に沿わせる。
- 安全（hibiki-security の該当トリガー・git前3ステップ）はこの緩和の対象外。常に遵守。
