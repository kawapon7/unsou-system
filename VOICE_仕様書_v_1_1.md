## 1. 概要・目的

### 1-1. コンセプト
運転中・現場作業中・両手がふさがっている状況でも、スマートフォンのマイクに話しかけるだけでシステムの操作・データ入力を完結させる「ハンズフリー業務入力」オプション。

子分（ドライバー）が走行中でも記録を残せるようにすることで、入力遅延・漏れをゼロに近づけ、月末の記憶頼り入力を根絶する。

### 1-2. ターゲットユーザー

| ユーザー | 主要ユースケース |
| :--- | :--- |
| ドライバー（driver UI） | 稼働終了時に音声で「今日の記録を登録して」「高速代1,200円を立替に追加」 |
| 管理者（admin UI） | ハンズフリーでダッシュボード確認・画面遷移 |

### 1-3. 設計原則
* HIBIKIコアのDB・ロジック・RLSポリシーを**一切改変しない**疎結合設計
* 音声認識処理はフロントエンド（ブラウザ Web Speech API）で完結させ、テキスト化されたインテントのみをHIBIKI APIへ送信する
* 音声データ（音声ストリーム）をサーバーへ送信・保存しない（プライバシー保護）
* HIBIKIコアのセキュリティ要件（AES-256-GCM暗号化・RLS全面遮断）を完全継承

## 2. 機能要件

### 2-1. 機能一覧

| 機能ID | 機能名 | 概要 | 優先度 |
| :--- | :--- | :--- | :--- |
| V-01 | 画面遷移コマンド | 音声コマンドで画面・タブを切り替える | 必須 |
| V-02 | 勤務記録音声入力 | 音声でフォームフィールドを自動補完・登録 | 必須 |
| V-03 | 立替金音声入力 | 音声で立替金データを登録 | 必須 |
| V-04 | 案件・取引先情報サジェスト | 案件名や取引先名を音声入力し候補をサジェスト | 推奨 |
| V-05 | ダッシュボード読み上げ | 今月のサマリーを音声で読み上げ | 推奨 |
| V-06 | 確認・キャンセルコマンド | 「登録して」「やっぱりやめて」で送信・取消 | 必須 |

### 2-2. V-01: 画面遷移コマンド

#### 対応コマンド一覧

| 発話例 | 遷移先 |
| :--- | :--- |
| 「ダッシュボードを開いて」「ホームに戻って」「案件一覧」 | `/admin/dashboard` |
| 「売上管理を開いて」「請求書を見せて」 | `/admin/sales` |
| 「取引先を開いて」「荷主一覧」「委託先一覧」「ドライバー一覧」 | `/admin/partners` |
| 「請求・支払を見せて」「支払通知書」 | `/admin/billing` |
| 「案件・配車管理」「配車を見せて」 | `/admin/projects` |
| 「今日の記録」「稼働を入力したい」 | `/driver/dashboard` |
| 「予定を見せて」「スケジュール」 | `/driver/schedule` |
| 「立替金を入力」「高速代を追加したい」 | `/driver/dashboard`（立替金セクションへスクロール） |
| 「ログアウトして」 | ログアウト処理 |

#### 実装フロー
```
Web Speech API（音声→テキスト）
  → クライアントサイドのキーワードマッチ
  → Next.js Router.push()による画面遷移（APIコール不要）
```

### 2-3. V-02: 勤務記録音声入力

#### 発話フロー例
```
ユーザー：「今日の稼働を登録したい」
VOICE   ：「案件名を教えてください」
ユーザー：「城南エリア宅配」
VOICE   ：「城南エリア宅配便でよろしいですか？」（候補をUI表示）
ユーザー：「はい」
VOICE   ：「開始時刻を教えてください」
ユーザー：「8時半から17時15分まで、休憩1時間、43個」
VOICE   ：「2026年6月7日・城南エリア宅配便・8:30〜17:15・休憩60分・43個で登録してよろしいですか？」
ユーザー：「登録して」
         → saveVoiceExpense / submitWorkRecord（Server Actions 経由）
```

#### 音声から抽出するフィールド

| フィールド | 認識パターン例 | フォールバック |
| :--- | :--- | :--- |
| `date` | 「今日」→当日 / 「昨日」→前日 | 手動選択 |
| `project_id` | 案件名の一部でファジーマッチ | プルダウン選択 |
| `start_time` | 「8時半」「8:30」「午前8時30分」 | 手動入力 |
| `end_time` | 「17時15分まで」「5時15分」 | 手動入力 |
| `break_minutes` | 「1時間」→60 / 「30分」→30 | 手動入力 |
| `quantity` | 「43個」「43件」→43 | 手動入力 |
| `note` | 「メモ：〇〇」以降の発話 | 任意 |

### 2-4. V-03: 立替金音声入力

#### 発話フロー例
```
ユーザー：「高速代を追加したい」
VOICE   ：「金額を教えてください」
ユーザー：「1,320円」
VOICE   ：「日付は今日ですか？」
ユーザー：「はい」
VOICE   ：「本日・高速道路料金・1,320円で登録してよろしいですか？」
ユーザー：「登録して」
         → saveVoiceExpense（`web/src/app/_actions/voice-actions.ts`）
```

#### 種別自動判定ルール

| 発話キーワード | 自動判定種別 |
| :--- | :--- |
| 「高速」「ETC」「有料道路」「圏央道」「首都高」 | `toll`（高速道路料金） |
| 「駐車」「パーキング」「コインパーキング」 | `parking`（駐車場代） |
| 「ガソリン」「燃料」「給油」 | `fuel`（燃料費補助） |
| 上記に該当しない | `other`（その他経費） |

### 2-5. V-04: 案件・取引先情報サジェスト

* 案件登録フォームで「城南エリア」と話すとマスタとファジーマッチし候補を表示
* 取引先登録フォームで「会社名を音声入力」し既存マスタと照合
* 音声入力後のフォーム内容は必ず視覚的確認ステップを挟む（音声のみでの確定禁止）

### 2-6. V-05: ダッシュボード読み上げ

* 「今月の状況を教えて」等のコマンドで、ダッシュボードのサマリーを音声合成（Web Speech Synthesis API）で読み上げる
* 読み上げ内容：入金予定総額・支払予定総額・粗利・未承認件数

## 3. システム連携要件

### 3-1. アーキテクチャ

```
[スマホブラウザ（driver / admin）]
  │
  ├── Web Speech API（音声→テキスト変換）    ← クライアント完結・音声データは外部送信なし
  │
  ├── VoiceButton コンポーネント              ← `web/src/components/voice/VoiceButton.tsx`
  │
  ├── VOICE Server Actions（インテント解析）  ← `web/src/app/_actions/voice-actions.ts`（Gemini 2.0 Flash）
  │
  └── Next.js Router.push()                  ← `/admin/*` または `/driver/*` へ画面遷移
        └── Supabase DB（RLS保護・service_role経由・Server Actions のみ）
```

### 3-2. HIBIKIコアAPIエンドポイント（利用するもの）

| エンドポイント / Server Action | メソッド | 用途 |
| :--- | :--- | :--- |
| `parseVoiceIntent`（`voice-actions.ts`） | Server Action | Gemini 2.0 Flash によるインテント解析 |
| `saveVoiceExpense`（`voice-actions.ts`） | Server Action | 立替金登録 |
| `submitWorkRecord`（`workRecordActions.ts`） | Server Action | 勤務記録登録 |
| `POST /api/hibiki/voice/intent` | POST | キーワードマッチ補助エンドポイント（レガシー互換） |

#### リクエスト共通ヘッダー
```
Authorization: Bearer {Supabase Session JWT}
Content-Type: application/json
X-Option-Source: voice/1.0
```

### 3-3. セキュリティ継承方針

| HIBIKIコア要件 | VOICEオプションの実装方針 |
| :--- | :--- |
| DB直接アクセス禁止 | VOICEモジュールはHIBIKI APIのみ呼び出し、Supabase clientを直接使用しない |
| 口座情報の暗号化 | VOICEオプションは口座情報の入力・表示機能を持たない |
| RLS全面遮断 | フロントからのSupabase直接クエリは一切行わない |
| 音声データ非保存 | 音声ストリームはブラウザ内でテキスト変換・外部送信禁止 |

## 4. 技術実装方針

### 4-1. 音声認識技術選定

| 技術 | 採用フェーズ | 理由 |
| :--- | :--- | :--- |
| **Web Speech API** | 初期（α・β） | ゼロコスト・追加インフラ不要・ブラウザネイティブ |
| **Whisper API** | 正式版（換装） | 高精度・ノイズ耐性・多言語対応（$0.006/分） |

初期はWeb Speech APIで低コスト展開し、認識精度が問題になった時点でWhisper APIへ**1箇所の換装**で切り替えられる抽象化レイヤーを設ける。

### 4-2. ファイル構成（v_1_1：admin/driver 移行後）

```
web/
├── src/
│   ├── app/
│   │   ├── _actions/
│   │   │   └── voice-actions.ts          # parseVoiceIntent / saveVoiceExpense
│   │   ├── admin/
│   │   │   └── shell.tsx                 # VoiceButton 配置（管理画面）
│   │   ├── driver/
│   │   │   └── shell.tsx                 # VoiceButton 配置（ドライバー用UI）
│   │   └── api/
│   │       └── hibiki/
│   │           └── voice/
│   │               └── intent/
│   │                   └── route.ts      # キーワードマッチ補助エンドポイント
│   └── components/
│       └── voice/
│           └── VoiceButton.tsx           # フローティングマイクUI（Web Speech API）
```

### 4-3. UIコンポーネント設計

```
[マイクボタン]（各フォームの右上に配置）
  → 押下：録音開始（赤点滅アニメーション）
  → 発話中：波形ビジュアライザー表示
  → 解析中：スピナー表示
  → 完了：抽出結果をフォームフィールドにプレビュー表示
  → 確認モーダル：「この内容で登録しますか？」
    → 「登録する」→ API送信
    → 「修正する」→ フォームを手動編集可能に戻す
```

## 6. リスクと対策

| リスク | 対策 |
| :--- | :--- |
| 走行中ノイズによる誤認識 | 確認ステップを必須化・手動修正を常に可能にする |
| iOS Safariのマイク権限制限 | PWAとして「ホーム画面に追加」を促し権限を安定させる |
| 業界用語（「城南」「ETC」等）の認識率低下 | Web Speech API Grammar でカスタム辞書を登録 |
| 個人情報（金額・案件名）の音声データ漏洩 | 音声ストリームはブラウザ内完結・サーバー送信禁止を徹底 |
