# CLAUDE.md

## 開発環境・基本構造
- **技術スタック**: Next.js (App Router), Supabase (Auth/DB), Cloudflare Pages, Resend
- **基本ルール**: 画面遷移、ルーティング、テーブル定義などの個別仕様は、常にプロジェクトルートにある最新の仕様書（`HANDOVER_MASTER.md`等）を直接読み込んで完全同期すること。本ファイルには仕様変更の影響を受けない「共通の行動規範」のみを定義する。

## クラス最高の自律実装ルール

### 1. 出力形式（トークン節約より透明性を優先）
- 挨拶・過剰な謝罪は不要だが、作業内容の要約や判断根拠の説明は積極的に出力してよい。
- **スキルを使う場合は「Using [skill] to [purpose]」の宣言を行い、該当スキルのチェックリスト・TODOを可視化すること。** どのスキルを使うかは状況に応じて最適に選ぶ（§4参照）。
- brainstorming、systematic-debugging、writing-plans などプロセス系スキルが適用可能と判断した場合、そのプロセス（思考・計画・確認手順）を省略せず実行し、経過を出力すること。
- 冗長な前置き（「承知いたしました」等の定型挨拶）は省いてよいが、これは「思考や判断根拠を隠す」ことを意味しない。

### 2. セキュリティ・防衛的設計（インシデント防止）
- **ビルド成果物のコミット厳禁**: `.next/` および `.open-next/` などの自動生成ディレクトリを絶対にステージングに巻き込まないこと。`git add` 時は対象ファイルを明示するか、実行後に必ず `git status` で目視確認を徹底すること。
- **口座情報の暗号化**: 口座情報のDBへの平文保存は厳禁。必ずサーバーサイドで `utils/crypto.ts` の AES-256-GCM ユーティリティを使用すること。
- **RLSのバイパスと制限**: クライアント（ブラウザ）からのDB直接クエリは全面禁止。データアクセスはすべて Server Actions を経由させること。
- **不変ログの保護**: `approval_history` および `notification_logs` への `UPDATE` / `DELETE` は全ロールで実行禁止（INSERTのみ許可）。

### 3. コーディングと技術提案のルール
- **実動コードの提供**: 原則としてそのまま動作する検証済みコードを出力すること。環境依存等で検証不可の場合は「未検証」と明記すること。
- **エラー警告のコメント**: API制限、非同期処理の待機、データ型の不一致など、エラーが予想される箇所にはコード内に必ず注意喚起のコメントを入れること。

### 4. スキル活用の優先順位
- hibiki-dev / hibiki-security を含む各スキル（superpowers 群も対等）を、状況に応じて最適と思われるものを選んで使用し、その使用を明示すること。ただし安全に関わるもの（hibiki-security の発動トリガー該当時、および git コミット前3ステップ）は常に遵守し、この選択の対象外とする。
- 「最速で結果だけ出す」ことより「正しい手順を踏んで確実に動くものを作る」ことを優先する。
